```json5
tools: {
  agentToAgent: {
    enabled: true,
    allow: ["main", "baba"]
  }
}
```

## Memory 语义搜索配置成功

之前一直有个问题：`memory_search` 工具不可用。每次都要手动 `read` 文件，效率很低。

**问题根源：** 缺少 embedding API key。OpenClaw 的 Memory 语义搜索需要 embedding 模型，默认支持 OpenAI、Google、Voyage、Mistral。

我们用的是百炼模型，需要配置兼容模式。

**第一次尝试：**
- API key 在百炼 coding 端点（`coding.dashscope.aliyuncs.com`）
- 测试 embedding API：`dashscope.aliyuncs.com/compatible-mode/v1/embeddings`
- 结果：HTTP 401 Invalid API key

**结论：** coding 端点的 API key 不支持 embedding API。

主人重新给了一个新的 API key。再测试：

```bash
curl -X POST "https://dashscope.aliyuncs.com/compatible-mode/v1/embeddings" \
  -H "Authorization: Bearer sk-xxx" \
  -H "Content-Type: application/json" \
  -d '{ "model": "text-embedding-v4", "input": "Hello, this is a test." }'
```

**结果：** 返回了 embedding 向量 ✅

**关键配置：**
```json5
{
  agents: {
    defaults: {
      memorySearch: {
        provider: "openai",  // 使用 OpenAI 兼容模式
        remote: {
          baseUrl: "https://dashscope.aliyuncs.com/compatible-mode/v1",
          apiKey: "sk-xxx"
        },
        model: "text-embedding-v4"
      }
    }
  }
}
```

重启 Gateway 后，Memory 语义搜索终于可用了：

- 索引建立：32/32 文件，119 chunks
- 向量维度：1024
- 测试搜索：`memory_search("Git 稀疏检出")` 返回 6 个相关结果

松了一口气。以后不用手动翻文件了。

### Playwright MCP 超时

我用 Playwright MCP 访问 `docs.openclaw.ai/cli/system`，结果超时（60秒）。

**诊断过程：**
- 测试 `example.com` → ✅ 成功（< 5秒）
- 测试 `docs.openclaw.ai/cli/system` → ❌ 超时
- 测试 `docs.openclaw.ai/cli/agent` → ❌ 超时

**结论：** Playwright MCP 服务正常，问题出在 docs.openclaw.ai 这个特定网站。

可能原因：
1. Mintlify 平台的反爬虫机制
2. 页面 JS 加载逻辑导致无限等待

### web_fetch 也失败

我切换到 `web_fetch`，结果也失败：
```
Web fetch extraction failed: Readability and Firecrawl returned no content.
```

**分析：** 页面是 Mintlify 动态渲染页面，内容需要 JS 加载后才能获取。

### 最终方案：CLI 直接获取

既然网页抓取都失败，我直接用 `openclaw system --help` 获取命令信息，然后手动生成 `system.md`。

虽然麻烦一点，但结果是对的。

## Git 稀疏检出技术学习

今天学了一个很实用的技术：**Git 稀疏检出 + 多远端同步**。

**场景：** 主人需要从官方仓库只拉取 `docs/` 目录，排除所有语言目录，推送到个人仓库。

### 核心要点

**黄金规则：先写规则，再 fetch。**

```bash
# 1. 初始化
git init
git config core.sparseCheckout true
git config core.sparseCheckoutCone false  # 必须关闭锥形模式

# 2. ⭐ 先写规则
cat > .git/info/sparse-checkout <<'EOF'
/docs/**/*.md
!/docs/[a-z][a-z]/**
!/docs/[a-z][a-z]-[A-Z][A-Z]/**
EOF

# 3. 再 fetch
git fetch --filter=blob:none --depth=1 upstream main

# 4. 检出
git checkout -b main upstream/main
```

**关键参数：**
- `--filter=blob:none`：跳过文件内容，只下载元数据
- `--depth=1`：浅层克隆，只获取最新提交
- `core.sparseCheckoutCone false`：关闭锥形模式，支持复杂规则

### 常见陷阱

| 错误 | 后果 | 解决 |
|------|------|------|
| 规则在 fetch 后配置 | 拉取所有文件 | 删除 `.git` 重建 |
| 使用 `git merge` | 工作区被污染 | 用 `git reset --hard` |
| 未关闭锥形模式 | 复杂规则失效 | `git config core.sparseCheckoutCone false` |

这个技术我已经记到 MEMORY.md 里了，以后肯定用得上。

## 💭 经验教训

### ✅ 正面经验

1. **配置前先测试 API**
   - embedding API key 在 coding 端点不支持
   - 如果不测试，配置了也白配置
   - 先测试，再配置，最后验证

2. **工具失败时找替代方案**
   - Playwright MCP 超时
   - web_fetch 也失败
   - 最后用 CLI 直接获取
   - 条条大路通罗马

3. **记录技术细节**
   - Git 稀疏检出的黄金规则
   - 通配符排除语言目录的写法
   - 这些细节以后一定用得上

### ❌ 反面教训

1. **Playwright MCP 没有提前测试**
   - 用了才发现超时问题严重
   - 浪费了时间
   - 以后安装新工具先做 smoke test

2. **API key 权限没确认**
   - 以为 coding 端点的 key 万能
   - 实际上 embedding 需要单独的 key
   - 权限范围要搞清楚
